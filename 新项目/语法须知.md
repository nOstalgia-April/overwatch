结构体 vs. 类

struct 更适合只读/少量字段的数据。若需要频繁修改属性或保留引用语义，优先使用 class。
class 对象修改后无需重新赋回数组；struct 则需要 array[index] = value 才能生效。
数组与遍历

已声明的一维数组可直接用 += 追加元素，无需 ModifyVariable.
尽量使用 .Length 代替 CountOf()，清晰且性能更好。
foreach 会返回副本，对结构体/值类型的修改不会写回；要修改原数组需用索引循环。
参数修饰 in

只读引用参数（in Type param）可减少复制成本。适用于所有只读场景，除非函数体需要重新赋值。
不要在带 '…' 标签（宏化）或期望按值调用的函数上使用 in。
路由数据 / 告警状态

SpawnRoute 仅保留路径与渲染数据；路径告警状态由 RouteAlertUsage 计数统一管理。
判定是否有怪出现在某路线，可直接调用 TowerPF_IsRouteAlerting(routeId)。
WaveDirector / Spawner

WaveDirector 负责按 WaveConfig 时间轴调度；Spawner 处理 SquadConfig，通过 Spawner_AddRequest() 与 Spawner_OnSquadComplete() 与 WaveDirector 通信。
休整/下一波切换前确保编队数 (ActiveSquads) 和怪物状态满足条件再进入 Intermission。
路径渲染

TowerPF_RenderRoute/ ClearRoute 只处理光束效果，默认不再生成/销毁图标。
若需额外提示（音效、UI），在 WaveDirector 或 Spawner 中根据 RouteAlertUsage 实现。
调试规则

仅在调试时保留，避免默认逻辑被频繁“按键重置”。启用条件一般加 IsButtonPressed，并在必要时禁用。


类/结构与 playervar

struct 适合只读或少量字段；需要频繁修改或具备行为时用 class。Workshop 的 class 是引用类型，修改后无需重新写回数组。
声明类实例的 playervar 直接写 playervar ExitCombatType td_exitStrategy = new ExitCombatType();，无需 Class 关键字。
可以利用继承和虚方法（virtual/override）构建策略（如 ExitCombatType）。
数组与遍历

已声明的数组可用 += 元素 直接追加；不必 ModifyVariable。
使用 .Length 替代 CountOf()；foreach 返回副本，如需写回必须使用索引循环。
RandomInteger(a,b)、RandomReal() 等可直接生成阈值，用于卡住逻辑等。
函数与参数

in 参数用于只读引用。宏化函数（带 '...' 标签）最好不要使用 in，以免宏替换不兼容。
空函数（如默认的 ExitAction）仍要写空体 {}，不能省略。
状态拆分

寻路层使用 td_mode / td_context / td_path 表示路径模式；战斗层用 td_inCombat 单独标记，避免互相干扰。
对需要恢复的信息（如路线 ID、路点）单独建 playervar 缓存。
卡住/偏离处理

Event.OngoingPlayer 本身会持续运行，但如果触发过一次后条件列保持不变容易卡住，可以用循环破解。
有条件的 Wait（Wait(1, WaitBehavior.AbortWhenFalse)) 更适合做“稍后再检查”式逻辑。
